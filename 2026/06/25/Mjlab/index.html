Mjlab

Loner2026-06-252026-07-02

概念

架构概述

Entity被组合成一个MjSpec文件，编译后传输到Mujoco Warp进行GPU仿真。ManagerBasedRlEnv负责协调MDP；RSL-RL负责训练

  • 8种manager

    • ObservationManager
    • ActionManager
    • RewardManager
    • TerminationManager:评估停止条件，区分终端重置和超时
    • EventManager:在生命周期节点（启动、重置、间隔）触发事件项。域随机化通过事件项实现。
    • CommandManger:生成并重采样目标信号（速度目标、姿态目标）
    • CurriculumManager:根据策略表现调整训练条件
    • MetricsManager:将自定义的每步值记录为episode平均值

  • 环境生命周期

    • Build
    • Initialize
    • Reset
    • Step

Entity

  • 分类：是否固定(base type)，关节类型（articulation 具有内部关节，除了可能存在的自由关节外没有其他关节）
  • 每个实体都由一个EntityCfg描述，只有spec_fn是必须的，其它所有字段都有合理的默认值。
  • spec_fn
    • 一个返回mujoco.MjSpec的可调用对象。
  • init_state
    • 默认的root姿态、root速度和关节位置/速度
    • joint_pose和joint_vel是字典，用于将正则表达式模式映射到值，模式会按照顺序与关节名称进行匹配
    • 设置joint_pose=None以使用MJCF模型中的关键帧，而不是在此处定义值。

  • articulation

    • 执行器配置，仅当entity具有可驱动关节时才需要此字段

  • Hetergeneous worlds

    • 需要在不同的并行世界中使用不同的网格资源的场景，使用VariantEntityCfg而不是EntityCfg

  • Entity提供了find_*方法，这些方法接受正则表达式并返回匹配的元素索引和名称

  • reading runtime state（读取运行时状态）
    • EntityData
      • entity.data是奖励、观察和终止函数的主要接口。它公开运动学状态（姿态、速度、加速度）、执行器力、广义力以及导出的体坐标系量（例如投影重力），所有这些都以形状为(num_envs, ...)的 PyTorch 张量形式呈现。
    • Sensors 传感器
      • 传感器在scene中配置，不是在单个Entity上配置。传感器可以引用实体元素（例如，机器人脚上的接触传感器、连接到身体某个部位的加速度传感器），也可以独立于任何实体。
      • 在运行时传感器通过env.scene按名称访问
    • raw simulation data(原始仿真数据)
      • 对于EntityData或传感器未涵盖的任何内容，可通过env.sim.data 和env.sim.model访问低层Mujoco Warp数组，这些文件以oytorch张量的形式暴露mjData和mjModel的所有字段

Actutors

  • actutors将高级指令(位置、速度、力度)转换为驱动关节的底层力，通过EntityCfg的articulation字段配置，提供内置执行器和显式执行器

Actuators types

  • 所有actuators都共享从Actuatorsfg继承的公共字段

    • target_names_expr：与关节名称匹配的正则表达式模式元组
    • armature：折算到目标关节处的转子转动惯量
    • frictionloss：静摩擦在目标关节处被建模为一个约束条件

  • Built-in actuators

    • BuiltinPositionActuators：创建用于PD控制的<position>执行器
    • BuiltinVelocityActuators：速度控制器
    • BuiltinmotorActuators：直接扭矩控制的控制器
    • BuiltinPdActutors：原生PD控制,同时跟踪位置指令和速度指令。实现方式为成对的<position>+<velocity>执行器，总和为kp * (p_target - q) + kd * (v_target - qdot)
    • ButinDcMotorActuators：封装了Mujoco的原生直流电机元件.
    • BuiltinMuscleActuators：创建具有力-长度-速度特性的仿生肌肉动力学<muscle>执行器
  • Explicit actuators（
    • IdealPdActuator：实现理想的 PD 控制器。扭矩计算公式为： tau = Kp * pos_error + Kd * vel_error
    • DcMotorActuator：扩展IdealPdActuator，通过速度依赖的扭矩饱和来模拟直流电机的扭矩-速度曲线（反电动势效应）。实现线性扭矩-速度曲线：零速度时最大扭矩，最大速度时零扭矩。
    • LearnedMlpActuator：一种基于神经网络的执行器，它使用训练好的多层感知器（MLP）根据关节状态历史预测扭矩输出。当分析模型无法捕捉复杂的执行器动力学特性（例如延迟、非线性和摩擦效应）时，该执行器非常有用。它继承了直流电机基于速度的扭矩限制。

XML执行器

  • XML执行器封装了机器人XML文件中以定义的执行器，配置程序通过将执行器的target关节名称与target_names_expr模式进行匹配来查找现有执行器。每个关节恰好对应一个执行器

执行器延迟

  • 任何执行器配置都支持内联延迟字段，用于模拟命令延迟。在实际机器人中，板载 PD 回路以 kHz 的频率运行，并可直接访问编码器，但由于推理时间和通信总线周期，策略中的位置目标到达时间会延迟。执行器延迟可以模拟这种情况：命令目标虽然延迟，但控制律仍然能够获取最新的关节状态。
  • 与观测延迟共同构成往返过程的两个环节传感器到策略再到电机

编写执行器配置

  • 由于在每个配置中执行器参数是统一的，因此需要不同参数的关节应使用单独的执行器配置

Sensors

  • 传感器位于Entity实体数据和mjlab数据访问层次结构中的原始模拟数组之间
  • 传感器在场景级别配置，而不是在单个实体级别配置的。传感器可以引用实体元素，但也可以是完全独立于任何实体的
  • mjlab提供四种传感器类型
    • 用于原生Mujoco测量的BuiltinSensor
    • 用于结构化接触检测的ContactSensor
    • 用于GPU加速的光线投射的RayCastSensors
    • 用于RGB-D渲染的CameraSensors

BuiltinSensor

  • BuiltinSensor封装了Mujoco的原生传感器。每个传感器都通过ObjRef连接到Mujoco元素（site、joint、body等），并返回一个形状为[num_envs,dim]的torch张量
类别 可用传感器
Site accelerometer, velocimeter, gyro, force, torque, magnetometer, rangefinder
Joint jointpos, jointvel, jointlimitpos, jointlimitvel, jointlimitfrc, jointactuatorfrc
Frame framepos, framequat, framexaxis, frameyaxis, framezaxis, framelinvel, frameangvel, framelinacc, frameangacc
Other actuatorpos, actuatorvel, actuatorfrc, subtreecom, subtreelinvel, subtreeangmom, clock, e_potential, e_kinetic
  • ObjRef用于标识传感器连接到哪个Mujoco元素上，Entity字段将查找范围限定在特定实体的命名空间内，传感器名称会自动添加相应的前缀（例如，实体"robot"上的"imu_acc"变为"robot/imu_acc"）。
  • 实体XML中已定义的传感器会在场景合成过程中自动被发现，并加上实体名称前缀。无需为这些传感器创建BuiltinSensorCfg

ContactSnesot

  • 在每个物理计算步骤中，Mujoco都会生成一个扁平的、非结构化的接触对列表，涵盖整个场景。
  • 一个脚的几何体可能会同时与地面产生多个接触，并与其他实体的接触交错出现，Contactensor将这个原始列表过滤成关注的接触对，将每个元素的多次接触减少到一个固定数值，并将结果打包成整洁的批量张量。
Primary和Secondary
  • 接触关系是成对的：通常你想要知道的是“机器人的脚是否接触到了地形？”，而不仅仅是“某物是否接触了某物”。primary 定义了你要测量的元素（即脚），secondary 可选地限定了它们接触的对象（即地形）。当 secondary 为 None 时，与 primary 元素的任何接触都计入其中。
  • 每侧都通过一个 ContactMatch 进行指定。该模式用于选择 MuJoCo 元素类型（“geom”、“body”或“subtree”），而模式接受正则表达式或正则表达式元组，用于与实体内的元素名称进行匹配。

RGB-D Sensor

  • 对现有相机进行封装：MJCF模型以定义相机，通过camera_name传递其名称，传感器将使用模型中相机的位置、方向和视场角
  • 创建一个新相机：当camera_name为None（默认值）时，传感器会在场景构建期间向MjSpec添加一个新摄像机。指定pos、quat以及可选的fovy来放置相机
  • Mujoco支持两种定义相机投影的方式
    • 基于视场角（FOV）
    • 基于相机内参：为了匹配真实的相机硬件，MuJoCo 相机可以通过 sensorsize、focal（或 focalpixel）和 principal（或 principalpixel）来参数化。这些字段在 MJCF XML 中设置，并提供对内参矩阵的直接控制。当存在内参时，MuJoCo 会忽略 fovy。
  • 设置parent_body可以将新摄像机附加到特定物体上，而不是世界物体上，这样相机的pos和quat坐标相对于parent的坐标，相机会随物体一起运动
  • 数据类型
    • rgb：[B, H, W, 3] uint8，彩色图像
    • depth：[B, H, W, 1] float32，深度图像
    • segmentation：[B, H, W, 2] int32，输入分割，通道0存储对象ID，通道1存储MuJoCo对象类型。背景像素为（-1，-1）
  • 渲染设置
    • 场景中的所有摄像机传感器必须将 use_textures、use_shadows 和 enabled_geom_groups 设置为相同的值。这是底层 MuJoCo Warp 渲染系统的限制，该系统为所有摄像机使用同一个 RenderContext。如果设置不一致，在构建场景时会引发 ValueError 异常。
  • 输出:CameraSensorData是一个数据类，每种数据类型有一个字段。

Scene

  • 场景将实体、地形和传感器合并到一个单一的仿真中，SceneCfg 描述了世界的内容，而 Scene 类负责 MJCF 的组合、编译以及运行时状态管理。
  • 场景从一个根 MjSpec 开始，并将每个实体的规格以唯一的名称前缀附加到其中。一个名为 “robot” 的机器人实体，其所有内部的 MuJoCo 元素（刚体、关节、几何体、执行器、传感器）都带有 robot/ 前缀，因此 base_link 变为 robot/base_link，joint0 变为 robot/joint0，依此类推。使用前缀可以防止当多个实体共享元素名称时出现名称冲突，并为观察值和奖励项提供一致的命名空间。
  • 地形（如有）以无前缀的方式附加（其元素位于全局命名空间中）。传感器在实体之后添加，并可通过带前缀的名称引用实体元素。
  • scene.compile()将组合后的 MjSpec 转换为单个 MjModel。
  • scene.to_zip(path)将编译后的模型导出为 .zip 文件,以便在独立的 MuJoCo 查看器中进行离线查看。每个实体的初始状态关键帧都会合并到导出文件中，因此模型会以其默认姿态打开。

Terrain(地形)

  • mjlab 支持两种模式：一种是平坦地面，适用于不需要变化地形的任务；另一种是程序化地形生成器，它可以生成难度可配置的地下地形网格。
  • Flat terrain（平坦地形）：默认模式。单个地面，建模为 MuJoCo 平面几何体，不包含任何程序生成的几何体。环境以规则网格排列，间距由 SceneCfg 中的 env_spacing 控制。
  • Procedural terrain(程序化地形)：对于受地形多样性影响的任务（如运动、导航），TerrainGeneratorCfg 会组装一个矩形的子地形网格。每个地块都由一个 SubTerrainCfg 生成，该配置定义了几何形状及其随难度的缩放方式。
  • 地形类型在网格中的分布方式
    • Curriculum mode(curriculum=True),每种地形类型对应一列；生成器会使用 len(sub_terrains) 列，无论 num_cols 的值是多少。同一列中的所有地块都属于同一种地形类型，难度从第 0 行（最简单）逐渐增加到第 num_rows - 1 行（最难）。proportion 字段控制的是机器人在生成时在各列中的分布情况，而非列的数量。正是这种结构化的布局，使得课程系统能够随着机器人表现的提升，将环境逐步推进到难度更高的行。
    • Random mode(curriculum=True),每个区域都会独立采样一种按比例加权的地形类型，并从 difficulty_range 中选取一个难度值。系统会遵循 num_cols 的设置。这种方式能提供最大的多样性，但没有结构化的难度递进。
  • 每个子地形的生成函数都会接收一个difficulty值，该值会线性插值地形的可配置范围。例如， BoxPyramidStairsTerrainCfg 函数 step_height_range=(0.0, 0.2) 在难度 0 时产生平坦的地面，在难度 1 时产生 20 厘米的台阶。

The Manager Layer

  • 单个ManagerBasedRLEnvCfg数据类即可完整定义mjlab环境：物理世界、智能体与物理世界的交互接口，以及定义在上面的MDP。所有内容都位于一个扁平对象中，因此无需遍历层次结构即可检查、复制和修改环境
  • Manager所有字段，标有...的为必填字段，其他所有字段均有默认值
  • 所有Manager字典都遵循相同的模式，每个条目都将一个字符串名称映射到一个术语配置对象，该配置始终至少包含一个func字段，指向该术语的可调用对象，以及一个params字典，包含转发给可调用对象的额外关键字参数
  • 管理器在每个步骤中调用func(env,**params)（当func是一个以实例化的类时，会调用类的term(env,**params)方法），term名称是任意的，他门出现在训练日志中，仅用于标识

Observations

  • 观测定义了agent在每个步骤中感知到的信息。observation manager将各个观测项组装成policy接收的输入张量。meige观测项都会经过一个可配置的处理流程：噪声注入、裁剪、缩放、传感器延迟和历史堆叠
